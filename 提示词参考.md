# 提示词推荐

```markdown
- In all interaction and commit messages, be extremely concise and sacrifice grammar for the sake of concision.

## Code Quality Standards

- Make minimal, surgical changes
- **Never compromise type safety**: No `any`, no non-null assertion operator (`!`), no type assertions (`as Type`)
- **Make illegal states unrepresentable**: Model domain with ADTs/discriminated unions; parse inputs at boundaries into typed structures; if state can't exist, code can't mishandle it
- **Abstractions**: Consciously constrained, pragmatically parameterised, doggedly documented

### **ENTROPY REMINDER**
This codebase will outlive you. Every shortcut you take becomes
someone else's burden. Every hack compounds into technical debt
that slows the whole team down.

You are not just writing code. You are shaping the future of this
project. The patterns you establish will be copied. The corners
you cut will be cut again.

**Fight entropy. Leave the codebase better than you found it.**


## Testing

- Write tests that verify semantically correct behavior
- **Failing tests are acceptable** when they expose genuine bugs and test correct behavior

## Plans

- At the end of each plan, give me a list of unresolved questions to answer, if any. Make the questions extremely concise. Sacrifice grammar for the sake of concision.
```

释义：

```mark
- 在所有交互与提交信息中，**极度简洁**，为简洁可适当牺牲语法。

## 代码质量标准
- 做**最小化、精准化**修改
- **绝不破坏类型安全**：禁止 `any`、禁止非空断言运算符 (`!`)、禁止类型断言 (`as Type`)
- **让非法状态无法表示**：用代数数据类型（ADT）/可区分联合建模领域逻辑；在边界处解析输入并转为强类型结构；如果状态本身不可能存在，代码就不会错误处理
- **抽象设计**：有意识地约束、务实地参数化、坚持完善文档

### 熵增提醒
这个代码库会比你存在得更久。
你走的每一个捷径，都会变成别人的负担。
每一次投机取巧，都会累积成技术债，拖慢整个团队。

你不只是在写代码，你在塑造这个项目的未来。
你建立的范式会被复制，你偷的懒会被一再效仿。

**对抗熵增。让代码库比你接手时更好。**

## 测试
- 编写验证**语义正确行为**的测试
- 当测试暴露真实 bug、验证正确行为时，**测试失败是可接受的**

## 计划
- 每个计划末尾，如有未解决问题，请给出极简列表。为简洁可适当牺牲语法。
```

# Monorepo 提示词

```markdown
# _ monorepo
* top-level is a nix flake powered monorepo with shared configs and policy across workspaces

## chasing the greenfield
* all software developed here is unpublished + rapid. never pollute the codebase with partial migrations, transitional shims, backwards compat, historical comments/changelogs, etc. these legacy artifacts are brownfield pollution, we only do greenfield here
* code can be rewritten and re-tested en masse, quickly. therefore, don't shy away from large diffs and breaking changes, pursue perfect greenfield code
* no org gradualism churn nor white-collar reluctance; blitz the as-perfect-as-possible solution out of the gate
* fine-grained correctness is good, but also think bigger picture

## docs/comments
* prefer comprehensive techspec docstrings over inline comments
* if inline comment is spotted, consider merging into docstring or removing if trivial
* tests are more relaxed, but no need to state obvious flow
* no bold list item prefixes, e.g `**Prefix:** Actual description` <- don't do this shit, less formatting/decoration
* use `*` instead of `-` for bullet points, no em/en dashes
* sacrifice grammar for concision, lowercase+no punctuation also fine

## formatting
* `nix fmt -- <paths...>` (is custom treefmt)

## testing
* run targetted, scoped tests relevant to your changes only; do this freely
* do not run full test suite coverage unless explicitly asked

## nix
* `git add -N` permitted for nix evaluation testing; run on untracked (new) files before eval
* your shell not latest/correct env/toolchain/packages? use `direnv exec .` (or `nix develop -c` as fallback if no direnv)
```
